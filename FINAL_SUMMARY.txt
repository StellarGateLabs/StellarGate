AMOUNT CANONICALIZATION - FINAL IMPLEMENTATION SUMMARY
================================================================================

ISSUE FIXED
-----------
Amount strings "10.00", "10.0", and "10" were returned exactly as sent by
clients, causing string-based comparisons to fail for equivalent values.

Location: src/api/payments.rs:333-347 (to_json function)
Problem: Amount was echoed back without canonicalization

SOLUTION IMPLEMENTED
---------------------
Three-layer canonical amount serialization:

1. WRITE LAYER (src/db.rs::create_payment)
   - Parse amount to stroops: parse_stroops("10.50") → 105,000,000
   - Convert back to canonical: stroops_to_string(105,000,000) → "10.5"
   - Store canonical form in database
   → Prevents non-canonical amounts from entering database

2. READ LAYER (src/api/payments.rs::to_json)
   - Defensively canonicalize amount field on JSON serialization
   - Applied to all response endpoints: create, get, list
   - Also canonicalizes paid_amount field
   → Ensures all HTTP responses return canonical forms

3. WEBHOOK LAYER (src/webhook.rs::build_payload)
   - Canonicalize amount, paid_amount, and delta fields in webhook payloads
   - Applied to all webhook event types
   → Ensures webhook recipients always see canonical amounts
